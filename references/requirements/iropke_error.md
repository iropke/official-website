# IROPKE Error Page Definition

---

## 1. Page Role

The error page is not just a fallback screen.  
It must provide:

- clear error recognition
- accurate status communication
- a calm and slightly witty recovery experience
- a direct path back to the homepage

The page should reduce frustration while maintaining trust.

---

## 2. Core Positioning

### One-line Definition
A structured error page that explains what happened clearly, responds with light wit, and helps users recover immediately.

### Tone & Direction
- Clear and accurate
- Calm, not overly apologetic
- Slightly witty, but never childish
- Helpful and reassuring

### Important Rule
The page must **not include the global header**.  
It should feel isolated, focused, and intentional.

---

## 3. Layout Structure

1. Error Status Area
2. Witty Message
3. Short Description
4. Recovery Actions
5. Optional Technical Hint
6. Footer-only minimal structure (optional)

---

## 4. Section Details (English Copywriting)

---

### 4.1 Error Status Area

**Purpose**
Display the exact error code and type clearly.

**Required Elements**
- Error code (e.g. 404, 403, 500, 503)
- Error title
- Visual emphasis

**Examples**
- 404 / Page Not Found
- 403 / Access Denied
- 500 / Internal Server Error
- 503 / Service Unavailable

---

### 4.2 Witty Message

**Purpose**
Add a human tone without reducing seriousness.

**Guideline**
The witty line should feel empathetic and intelligent, not comedic.

**Example Messages by Error Type**

#### 404
“I understand that feeling too. Getting lost happens.”

Alternative:
“Even good paths sometimes disappear.”

#### 403
“This door seems to require a different key.”

Alternative:
“You’ve reached a page that prefers introductions first.”

#### 500
“Something behind the curtain seems to be having a moment.”

Alternative:
“The system blinked first.”

#### 503
“We’re briefly unavailable. Even systems need a breath.”

Alternative:
“This service is taking a short pause before returning.”

---

### 4.3 Short Description

**Purpose**
Explain the issue in plain language.

**Examples**

#### 404
The page you were looking for may have been moved, renamed, or no longer exists.

#### 403
You do not currently have permission to access this page.

#### 500
Something went wrong on our side while processing your request.

#### 503
The service is temporarily unavailable. Please try again in a few moments.

---

### 4.4 Recovery Actions

**Purpose**
Give users a clear next step.

**Required Buttons**
- Home
- Optional: Go Back
- Optional: Contact Us / Project Inquiry

**Primary Requirement**
The **Home** button must always be visible and easy to identify.

**Recommended Button Labels**
- Home
- Go Back
- Contact Us

---

### 4.5 Optional Technical Hint

**Purpose**
Provide useful extra information when appropriate, without overwhelming users.

**Examples**
- Error Code: 404
- Request could not be completed
- Please contact support if the issue continues

**Rule**
Technical hints should remain secondary and visually subdued.

---

## 5. Required UI Components

- No global header
- Large error code display
- Error title
- Witty message line
- Plain-language description
- Primary Home button
- Optional secondary actions
- Minimal layout with strong readability

---

## 6. Message Flow Summary

1. The user recognizes the error immediately
2. The page explains the issue clearly
3. A light human message reduces friction
4. The user sees an obvious recovery action
5. The user can quickly return to Home

---

## 7. Error Types to Support

### 404
Page Not Found

### 403
Access Denied

### 500
Internal Server Error

### 503
Service Unavailable

Optional additions:
- 400 Bad Request
- 401 Unauthorized
- 429 Too Many Requests

---

## 8. UX Notes

- Do not use the standard site header
- Centered layout is recommended
- Keep the content compact and vertically balanced
- Error code should be the strongest visual element
- The witty message should appear directly below the title
- The Home button should be visually primary
- On mobile, ensure buttons are stacked and easy to tap

---

## 9. Recommended Visual Direction

- Minimal and clean
- Calm spacing
- Subtle illustration or symbolic graphic allowed
- Avoid loud warning styles unless the error is security-related
- The tone should feel intelligent and composed
